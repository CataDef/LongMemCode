# LongMemCode v0.2 — methodology

> **TL;DR.** v0.2 extends v0.1 with four workflow-shape scenario
> categories that capture sustained-session behaviour, not just
> one-shot retrieval. All ground truth remains deterministic (no LLM
> judge). We are explicit about the oracle strength of each new
> category — see §4.

---

## 1. Why v0.2 exists

v0.1 of LongMemCode scored adapters on **task shape**: can you find a
symbol, complete a function, locate a bug? Real coding-agent traces
show that's only part of the workload. About half of all user prompts
in production sessions are not fresh investigation tasks — they are
**operations on prior context**: approvals, status checks, pronoun
references, redirects, repeated lookups across an investigation.

A memory system that nails every ApiDiscovery scenario but reloads the
entire repo context for every "ok" or "and how is it cancelled?" wears
out a token budget surprisingly fast. v0.2 measures that.

We did not invent these patterns. We observed them in a sample of 933
user prompts across three production developer sessions on three
different stacks (Remix / Next.js / Rust). The four new categories
described in §3 are direct expressions of patterns we counted there.

---

## 2. What v0.2 changes vs v0.1

- **Adds 4 categories** (§3).
- **Reweights the suite** so workflow-shape categories take roughly
  half of the total scoring mass. v0.1 categories remain present and
  reusable.
- **Adds a new corpus packaging**: `scenarios/kubernetes-v2.json`,
  built directly from the live Kubernetes v1.32.0 corpus using both
  v0.1 oracles (scip-go round-trip) and a new git-history mining
  pipeline (§3.4).
- **Documents oracle strength explicitly** (§4). v0.1 oracles are
  exact-set / contains-required. Some v0.2 categories use a weaker
  "non-empty result is pass" oracle, called out below; v0.3 will pin
  exact expected results for these.
- **No protocol changes** — adapters that speak the v0.1
  [adapter protocol](ADAPTER_PROTOCOL.md) consume v0.2 scenarios
  unchanged.

The v0.1 corpora and scenarios remain valid and continue to be
scored; v0.2 is additive.

## 3. The four new categories

### 3.1 ConversationalContinuation

**What it tests.** Whether the memory system can serve a follow-up
query cheaply when prior context is warm. The follow-up syntactically
depends on the prior query (pronoun reference, approval phrase,
redirect to a related symbol).

**How scenarios are generated.** Take a v0.1 scenario whose `query`
resolves to a single concrete symbol. Emit it as the
`preceding_query` (to be replayed by the runner first to warm the
adapter). Then attach a `query` that operates on the same symbol but
asks something different: callers, callees, or a bare-name lookup.

**Sub-types** (in the JSON):

- `pronoun_continuation` — follow-up is a `callers` query
- `approval_followthrough` — follow-up is a `lookup` (bare name)
- `redirect_to_callees` — follow-up is a `callees` query

**What a passing adapter does.** Returns the right symbols for the
follow-up. Adapters that re-fetch context per call (LLM-backed memory
systems) take a noticeable latency hit; structural adapters do not.

**Sample size in `kubernetes-v2.json`**: 261 scenarios.

---

### 3.2 RereadCascade

**What it tests.** Whether the memory system services repeated reads
of the same symbol within one session at the same cost as the first
read. Real traces show single source files being read 40-120 times
per session. A memory system that doesn't cache pays N× the cost.

**How scenarios are generated.** Take a v0.1 `lookup` scenario with a
concrete name. Repeat the same query 3-5 times in a `queries[]`
array. Single-query runners can use the `query` field which carries
the same payload.

**Sample size**: 109 scenarios.

---

### 3.3 SubAgentContextHandoff

**What it tests.** Whether the memory system serves a fresh process
the same view of the corpus that the parent process has. Real coding
agents delegate "explore X" to sub-agents that don't inherit session
state.

**How scenarios are generated.** Pair scenarios from v0.1: the
`main_query` is a `file_symbols` overview; the `subagent_query` (and
the runner's primary `query`) is a `callers` drill-down on a specific
symbol from that file's surface.

**What a passing adapter does.** Returns the same answer the parent
would have. Adapters with per-process state (e.g. unbounded LLM
context windows that never persist anything) fail this category by
construction — that's the intent.

**Sample size**: 50 scenarios.

---

### 3.4 BashFeedbackLoop

**What it tests.** Whether the memory system can resolve a
build/test failure to the source symbol that caused it, given only
the stderr line.

**How scenarios are generated — grounded in real Kubernetes git
history.** This is the most rigorous of the four:

1. Mine `git log --grep "^fix:" --grep "^bug:" --grep "^Fix bug" -i`
   over the full Kubernetes history (~127 000 commits at v1.32.0).
2. For each fix commit, take the first changed `.go` file.
3. Extract a Go function or type identifier from the current version
   of that file (the symbol the commit touched).
4. Synthesise a Go-style compiler error pointing at that symbol:
   `undefined: X`, `X redeclared in this block`,
   `cannot use X (untyped nil constant) as int value`,
   `X not declared by package`.
5. Ground truth: the symbol is the exact one the fix commit touched.
   The commit SHA is recorded in the scenario as
   `context.rationale_commit` for reproducibility.

This is the strongest oracle in the v0.2 suite — every scenario maps
back to a real commit you can `git show` to verify.

**Sample size**: 100 scenarios.

## 4. Oracle strength — be honest

LongMemCode has always avoided LLM judges; v0.2 keeps that. But not
all deterministic oracles are equally strong. Reviewers should weigh
these accordingly.

| Category                    | Oracle                  | Strength | Notes |
|-----------------------------|-------------------------|----------|-------|
| BugFix, Completion, ApiDiscovery, Refactor, FeatureAdd, TestGen (v0.1 rollover) | scip_roundtrip (exact_symbol / exact_set / contains-required) | **strong** | An adapter must return the exact stable IDs. |
| BashFeedbackLoop            | git_fix_commit_grounded + bare-name lookup | **strong** | The bare name maps deterministically to a real commit's touched symbol. |
| ConversationalContinuation, RereadCascade, SubAgentContextHandoff | non-empty result via `kind: contains, required: []` | **WEAK — v0.3 will pin** | Any non-empty response is currently scored as pass. This catches "you didn't crash" but not "you returned the wrong symbol." |

The weak-oracle categories are still useful in v0.2 for measuring
**latency and cost** under workflow-shape access patterns — that's
where adapters with read-time LLM hops bleed cost. But a v0.2
accuracy number on the weak-oracle categories should be read as
**"the adapter responded coherently with non-empty results,"** not
"the adapter returned the canonically correct results."

**v0.3 plan**: oracle-run a reference adapter (ArgosBrain) offline to
compute the canonical expected stable IDs for each weak-oracle
scenario, then pin those into the JSON. Once pinned, the same
scenarios become exact-set tests. Adapters that return *different
non-empty results* will then fail rather than pass. v0.3 will be
released as a strict superset of v0.2 with stronger oracles, no
scenario churn.

## 5. Reproducibility

The generator script lives at
[`tools/generate_v2_kubernetes_scenarios.py`](../tools/generate_v2_kubernetes_scenarios.py).
It is deterministic given a fixed `random.seed(42)`. To regenerate
the suite from scratch:

```bash
# 1. Build / pull the Kubernetes corpus (clones + scip-go + bundle)
./corpora/kubernetes.sh

# 2. Unshallow git history so BashFeedbackLoop can mine fix commits
( cd corpora/_work/kubernetes/source && git fetch --unshallow --tags )

# 3. Run the generator
python3 tools/generate_v2_kubernetes_scenarios.py
#   → writes /tmp/lmc2_gen/kubernetes-v2.json

# 4. Build adapter + runner
cargo build --release --bin lmc-runner --bin lmc-adapter-argosbrain

# 5. Run
./target/release/lmc-runner \
    --adapter ./target/release/lmc-adapter-argosbrain \
    --adapter-args "--corpus corpora/_work/kubernetes/kubernetes.argosbundle" \
    --scenarios /tmp/lmc2_gen/kubernetes-v2.json \
    --out results/your-adapter-kubernetes-v2-YYYY-MM-DD.json
```

Bundle id of the published reference run:
`a38adae682e89f9ec86b59373c2b57cbd85d3aadc29bcedc9a5fe9beea506008`

## 6. Positioning

LongMemCode v0.2 measures **code-structural memory** under sustained
coding-agent workloads. Adapters in different scope categories
(general conversational memory like Mem0 / Zep / Letta; pure semantic
embeddings like sbert-faiss; structural-graph adapters like
ArgosBrain) will all run on the suite and produce useful data, but
their numbers are not meant to rank them as substitutes for one
another. **General-purpose memory and code-structural memory are
complementary stacks.** The scoreboard should be read with that in
mind; the per-category breakdown often makes scope mismatch obvious.

A production agent typically benefits from running both classes of
memory at once. We benchmark them on the same suite so practitioners
can pick the operating point that matches their workload, not so we
can declare a winner.
