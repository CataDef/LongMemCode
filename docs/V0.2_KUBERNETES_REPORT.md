# LongMemCode v0.2 — Kubernetes at Scale, Four Adapters

*Published 2026-04-22.*

This release extends LongMemCode from the 5-corpus v0.1 set to a scale corpus
— **Kubernetes v1.32.0** — and adds two reference adapters beyond the
structural / text-search pair that shipped in v0.1:

- **`sbert-faiss 0.1.0`** — a dense-semantic baseline
  (sentence-transformers/all-MiniLM-L6-v2 + FAISS HNSW, symbol-level
  chunks keyed by SCIP stable id).
- **`mem0 2.0.0 @ gpt-4o-mini`** — the open-source general-purpose
  LLM-backed memory system by Mem0, configured with Chroma as the vector
  store and `gpt-4o-mini` as its extraction + search LLM.

The full adapter configurations are at the bottom of this report and inside
each `results/*.json` under the `adapter.config` key. The point of this
release is not a ranking — it is to make visible what happens when different
classes of memory system are asked the same code-structural questions on a
real production-scale corpus.

---

## A note on fairness, before the numbers

LongMemCode is a **code-structural-memory** benchmark. It measures retrieval
quality on the kinds of queries AI coding agents issue during inner-loop
work — *does this symbol exist*, *list methods of class X*, *who calls Y*,
*which function reads this config key*. Every scoring function in the
benchmark is a deterministic check against the source code's canonical-
identifier graph.

Two of the four adapters in this release are **not** code-structural memory
systems by design:

- `sbert-faiss` is a dense-retrieval baseline. It retrieves by vector
  similarity. It has no graph structure.
- `mem0` is a general-purpose memory system. Its intended workload is
  conversational memory — user preferences across sessions, multi-turn
  agent decisions, long-term user context. It is not a code-structural
  retrieval engine.

The measurement below tells us what happens when each system is asked
code-structural questions. It is **not a verdict on these systems in the
roles they were built for.** Mem0 on a conversational benchmark such as
LongMemEval or LoCoMo can beat a structural system trivially — because
those benchmarks measure the job Mem0 was built for.

We publish these numbers so:

1. Developers choosing a memory layer for their **coding** agent can see
   the real trade-offs on a realistic corpus.
2. Future code-memory research has a quantitative baseline to iterate
   against.
3. Any team that believes a different configuration better represents their
   system on code workloads can run it and submit — the adapter protocol
   is MIT and the scoreboard is an open venue.

Exact configurations, ingest logs, and failure modes are documented below
and in the per-result JSON files. Where a system hit a bug at our test scale
(Mem0 did — see §5), we report what we measured rather than retry until
numbers look clean.

## 1. Corpus and scenarios

- **Corpus**: [`kubernetes/kubernetes` v1.32.0](https://github.com/kubernetes/kubernetes),
  indexed with [`scip-go` 0.2.1](https://github.com/scip-code/scip-go) using
  `--skip-tests`. Project code only; vendored dependencies are not indexed.
- **Symbols**: 38 771 canonical SCIP identifiers across 1 283 project
  files, 3 244 Go packages.
- **Scenarios**: 1 456 generated via `lmc-scenario-gen --count 2000` (yield
  ~73 %) against the `kubernetes.argosbundle`. Distribution:
  - Completion: 468
  - BugFix: 292
  - ApiDiscovery: 256
  - FeatureAdd: 152
  - Refactor: 121
  - TestGen: 82
  - Config: 52
  - ControlFlow: 33
- **500-scenario stratified subset** (`scenarios/kubernetes-500.json`):
  drawn proportionally from the 1 456, used for the semantic and Mem0 runs
  to keep LLM costs reasonable. Categorical distribution is preserved
  exactly. Accuracy is within 0.1 % between 500-subset and full 1 456 for
  the adapters that ran both.

| Category     | ArgosBrain | grep   | sbert+FAISS | Mem0   |
|--------------|-----------:|-------:|------------:|-------:|
| Completion   |    97.65 % |  0.00 %|     18.63 % | 0.00 % |
| BugFix       |   100.00 % |  0.07 %|      6.00 % | 0.00 % |
| ApiDiscovery |   100.00 % | 39.06 %\**|    1.14 % | 35.23 %\** |
| FeatureAdd   |   100.00 % |  0.00 %|      1.92 % | 0.00 % |
| Refactor     |   100.00 % |  0.00 %|      0.00 % | 0.00 % |
| TestGen      |   100.00 % |  0.00 %|     35.71 % | 0.00 % |
| Config       |   100.00 % |  0.96 %|      0.00 % | 0.00 % |
| ControlFlow  |   100.00 % |  0.00 %|     54.55 % | 0.00 % |

\** ApiDiscovery contains 100 *adversarial* scenarios where the correct
answer is the empty set (fabricated identifiers that must NOT resolve).
Text search and LLM-backed systems score by accident on this half —
returning nothing is the right answer when the identifier does not exist.
The structural half of the category is where the real signal lives, and
both drop toward 0 % there.

The pattern matches the prediction in the companion paper
[*Structural vs Semantic Retrieval in Code-Memory: A Query-Type
Taxonomy*](https://argosbrain.com/papers/structural-semantic-taxonomy):
every category except Completion is definitionally structural (who calls
*X*, who overrides *Y*, list methods of *Z*), and semantic retrieval
collapses there because the question is "does this edge exist in the
graph", not "is there a vector near this query". Completion is the one
category where semantic retrieval has a handhold — unique symbol names in
source context embed into discriminable clusters — and sbert-faiss is
accordingly the only non-structural adapter to score double-digits there.

## 4. The grep-baseline caveat

grep's 5.48 % weighted average conceals its real behaviour. Without the
39 % accidental hit from adversarial scenarios, grep scores 0 % or
fractional percentages on every structural category. grep cannot return
canonical SCIP identifiers — it returns file paths and text matches — so
any scoring function that requires `exact_symbol` or `in_top_k` of stable
ids produces 0 for grep regardless of whether grep found the right line.
grep does the job it is named for; the job is not the job this benchmark
measures.

## 5. Mem0 — what we measured, what broke, and how we report it

We ran `mem0ai 2.0.0` with `gpt-4o-mini` as the LLM (the model Mem0's own
documentation uses in examples and that we judged representative of the
system's typical production configuration) and `text-embedding-3-small` as
the embedder, with `chroma 1.5.8` as the local vector store (Mem0's default
embedded backend).

**Ingest behaviour at our scale.** Under 32-worker parallelism (chosen to
keep ingest wall-clock reasonable on 58 311 symbols), Mem0's add path
produced a concurrent-write race in the Chroma backend: at 32 workers,
16 070 of 58 311 `add()` calls failed with `IndexError: index N is out of
bounds for axis 0 with size N`, a documented symptom of parallel writes
exceeding the backend's internal indexing throughput. An earlier attempt
with the Qdrant backend corrupted at 7 036 successful adds; Chroma sustained
the load for longer but still failed 27.6 % of adds over the full corpus.

We report the numbers we measured with 42 241 memories (72.4 % of the
corpus) indexed successfully. We could have re-run with fewer workers or
serialised adds to eliminate failures, but that would be a different
configuration; we publish what the configuration we ran produced. A future
submission from the Mem0 team (or anyone with knowledge of their tuning)
can correct this with a full-coverage ingest and a fresh number.

**Read-path latency.** P99 of 1.677 seconds reflects Mem0's architecture
— every `search` call involves an LLM round-trip for query understanding
plus a vector lookup plus LLM reranking. This is not a regression; this is
what Mem0 does on every read. LongMemCode's scoring target is interactive
IDE latency (< 30 ms is the pitch point for a tool call agents make 5-10
times per user turn), and the 1.7 s P99 is reported in that context —
not as a failure of Mem0, but as the natural consequence of running an
LLM-backed memory system inside the benchmark's IDE-inner-loop frame.

## 6. What we did not measure, and should

1. **Mem0 with a tuned code configuration.** Our run used production-common
   defaults. A tuned configuration — different LLM, bespoke extraction
   prompt for code, code-specific embedder, different vector store — may
   move the number. We publish this baseline as one data point; we claim
   no authority on "how Mem0 should be configured for code".

2. **Stronger semantic baselines.** `sbert-faiss` here uses MiniLM-L6-v2
   (384-d, CPU-friendly). A code-specific embedder
   (Jina-Code-v2, voyage-code, bge-code-large) would likely lift the
   semantic scores on Completion — the category where dense retrieval
   already has a handhold. Same submission invitation applies.

3. **Scale ceiling.** Kubernetes v1.32.0 project code indexes to ~38 771
   symbols. Kubernetes **including** vendored dependencies is on the order
   of 500 000 symbols. Linux kernel and Chromium class corpora are another
   order of magnitude above that. ArgosBrain's current numbers are bounded
   to the measured scale, as acknowledged in
   [Paper 1 §9.3](https://argosbrain.com/papers/longmemcode-benchmark).

4. **One machine, one OS.** All numbers collected on Apple M4 Max / macOS
   26.3, 48 GB RAM. Cross-machine variation (x86_64 Linux, Windows, tiny
   ARM) is not characterised in this release.

5. **Latency P99 at small n.** At n = 19 (exploratory runs) we measured
   ArgosBrain P99 at 165 ms due to cold-start contamination of the first
   query's percentile position. At n = 1 456 the cold-start artifact is
   diluted into the tail; P99 settles at 0.366 ms. Readers working with
   small scenario subsets should warm the adapter before measuring
   percentiles.

## 7. Adapter configurations — the exact knobs

All four JSON result files under `results/` carry a complete
`adapter.config` object. Summary:

### ArgosBrain 0.1.0
```
bundle: kubernetes.argosbundle (16 MB, 38 771 symbols, 232 756 edges)
embedder: Jina-Code-v2 INT8 (768-dim, quantised ONNX, local)
backend: petgraph in-process, bincode-serialised
read path: graph traversal + HNSW + keyword hybrid, no LLM
```

### grep-baseline 0.1.0
```
tool: ripgrep 14.1.0
scope: source tree walked from corpus root
response: file paths + match lines, mapped to stable ids only by file-path
          match — fails by construction on canonical-id scoring
```

### sbert-faiss 0.1.0
```
embedding_model: sentence-transformers/all-MiniLM-L6-v2 (384-dim, CPU)
index: FAISS IndexHNSWFlat, HNSW32, efConstruction=80, efSearch=64
chunking: symbol-level, ±6 lines of source context per definition
indexed_symbols: 58 311 (named SCIP definitions after skipping anonymous
                locals and non-gomod symbols)
top_k: 10
structural ops (callers/callees/file_symbols/etc): returned as empty set,
                honest; cannot be answered by embedding retrieval
```

### mem0 2.0.0 @ gpt-4o-mini
```
package: mem0ai 2.0.0 (PyPI)
llm: openai/gpt-4o-mini (default Mem0 production config)
embedder: openai/text-embedding-3-small
vector_store: chroma 1.5.8 (embedded, local)
ingest_workers: 32
ingest_success_rate: 42 241 / 58 311 = 72.4 %
ingest_failure_mode: Chroma concurrent-write race under 32-worker load
                     (bug in Mem0 + Chroma stack at this scale, not
                     benchmark; documented in §5 above)
query_path: memory.search(filters={user_id: k8s}, limit=10) with metadata
            stable_id mapping on return
top_k: 10
structural ops: returned as empty set, same as sbert-faiss — Mem0 has no
                graph primitive for callers/overrides/contained-by queries
```

## 8. Reproducing this release

```bash
# Prerequisites: Go 1.22+, Rust 1.85+, Python 3.11+, ripgrep 14+
git clone https://github.com/CataDef/LongMemCode.git
cd LongMemCode
cargo build --release

# Kubernetes corpus (clone + scip-go index + bundlegen)
bash corpora/kubernetes.sh

# ArgosBrain
./target/release/lmc-runner \
  --adapter ./target/release/lmc-adapter-argosbrain \
  --adapter-args "--corpus corpora/_work/kubernetes/kubernetes.argosbundle" \
  --scenarios scenarios/kubernetes-2k.json \
  --out results/argosbrain-kubernetes-$(date +%F).json

# grep-baseline (same flags pattern; --adapter-args --bundle … --source …)

# sbert-faiss
pip install sentence-transformers faiss-cpu numpy
python3 adapters/sbert-faiss/adapter.py \
  --corpus .../kubernetes.argosbundle --source .../kubernetes/source \
  --scip-json .../kubernetes.scip.json

# mem0 (requires OPENAI_API_KEY)
pip install mem0ai chromadb
export OPENAI_API_KEY=...
python3 adapters/mem0/adapter.py \
  --corpus ... --source ... --scip-json ... --user-id k8s
```

Exact scenarios: `scenarios/kubernetes-2k.json` and
`scenarios/kubernetes-500.json`, committed with stable SHA-256 recorded in
each result JSON.
