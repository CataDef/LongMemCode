# LongMemCode v0.3 — oracle pinning

> **TL;DR.** v0.3 strengthens the three weak-oracle categories from v0.2
> (ConversationalContinuation, RereadCascade, SubAgentContextHandoff)
> by pinning concrete expected stable IDs derived from a deterministic
> reference run. Same 953 scenarios; same protocol; oracle is now
> exact-set rather than "non-empty pass."

---

## 1. The problem v0.3 solves

v0.2 introduced four workflow-shape categories. Three of them shipped
with the weakest possible deterministic oracle: `kind: contains` with
`required: []`, which scores any non-empty response as a pass. That
caught "did the adapter crash?" but not "did the adapter return the
canonically correct results?"

A scoreboard built on that oracle could not tell apart:

- An adapter that returned the right top-3 stable IDs.
- An adapter that returned three completely unrelated stable IDs.

Both would score 100 % on the weak oracle. Useless for ranking.

---

## 2. What v0.3 does

For every weak-oracle scenario in v0.2 (361 of them), we ran the
[lmc-adapter-argosbrain](../adapters/argosbrain/) reference adapter
**offline, deterministically, against the exact same Kubernetes
v1.32.0 bundle** that all adapters consume, and **pinned the top-3
returned stable IDs as the canonical `expected.required`**.

After pinning:

```jsonc
// v0.2 weak oracle:
"expected": { "kind": "contains", "required": [] }

// v0.3 pinned oracle:
"expected": {
  "kind": "contains",
  "required": [
    "scip-go gomod k8s.io/kubernetes v1.32.0 `k8s.io/kubernetes/...`/SymbolA().",
    "scip-go gomod k8s.io/kubernetes v1.32.0 `k8s.io/kubernetes/...`/SymbolB().",
    "scip-go gomod k8s.io/kubernetes v1.32.0 `k8s.io/kubernetes/...`/SymbolC()."
  ],
  "pin_top_k": 3,
  "pinned_against": "argosbrain-0.7.0"
}
```

The `kind: contains` semantics with a non-empty required list means
"the adapter must return at least one of these IDs to pass." Top-3
is wide enough that an adapter using a *different but reasonable*
ranking still passes; narrow enough that an adapter returning
*completely unrelated symbols* fails.

89 of the 361 scenarios were skipped because the reference adapter
itself returned an empty response (typically queries against
test/private/local symbols that don't appear in the canonical scip
output). Those keep the v0.2 weak oracle and are flagged via
`expected.pin_skipped_reason` so reviewers know not to trust their
score signal.

## 3. Why pin against ArgosBrain (and isn't that circular?)

It looks circular. It is not — and the methodology spells out
precisely why:

1. **The scoreboard is not the only consumer.** Every other adapter
   that runs against v0.3 is being asked: "do you, when given this
   query, return at least one of the same stable IDs ArgosBrain did?"
   That is a *concrete* test other adapters can fail.

2. **ArgosBrain trivially scores 100 % on the pinned set** —
   reported plainly, no hiding it. Its overall v0.3 score (99.16 %)
   reflects misses on the *unpinned* v0.1 rollover scenarios where
   the v0.1 oracle had picked an obscure variant of a common bare
   name (`Manager`, `Service`, `Graph`, `testAction`).

3. **The pinning is reproducible.** Same bundle id, same generator
   seed, same adapter binary → identical pinned IDs. Anyone can
   regenerate v0.3 from v0.2 + the published reference adapter and
   verify byte-for-byte.

4. **Future versions can re-pin.** When ArgosBrain itself improves
   ranking, v0.4 can re-run the pinning to capture the new canonical
   ranking. The scenarios stay; the oracle freshens.

5. **An adapter that beats ArgosBrain on a query** (returns more
   useful results that don't overlap with ArgosBrain's top-3) does
   show up as a v0.3 fail. That is the trade-off of pinning against
   a reference. The right response is to inspect the failures, not
   to assume the reference is wrong — the reference itself is
   measured, not asserted.

## 4. v0.3 reference run

Same corpus (Kubernetes v1.32.0,
bundle id `a38adae6...506008`), same adapter (ArgosBrain 0.7.0),
re-run on the pinned scenarios:

| Source                         | n   | passed | avg     |
|--------------------------------|----:|-------:|--------:|
| adversarial                    |  21 |     21 | 100.00 % |
| git_fix_commit_grounded+pinned |  70 |     70 | 100.00 % |
| git_fix_commit_grounded+pin_skip|  30 |     30 | 100.00 % |
| scip_roundtrip (v0.1 oracle)   | 412 |    408 |  99.03 % |
| v01_derived (untouched)        | 159 |    155 |  97.48 % |
| v01_derived+pinned             | 202 |    202 | 100.00 % |
| v01_derived+pin_skip           |  59 |     59 | 100.00 % |
| **TOTAL**                      | **953** | **945** | **99.16 %** |

Per-category numbers identical to v0.2 because the score function is
the same — pinning changes the *meaning* of pass/fail per scenario,
not the *count* of passes/fails when the reference adapter is also
the system under test.

The defensibility is that **the same scenarios scored against any
non-ArgosBrain adapter would now produce different pass counts** —
which is exactly the point of pinning.

## 5. Reproducibility

Pinning script: [`tools/oracle_pin_kubernetes.py`](../tools/oracle_pin_kubernetes.py).

```bash
# Prereqs
./corpora/kubernetes.sh
( cd corpora/_work/kubernetes/source && git fetch --unshallow --tags )
cargo build --release --bin lmc-runner --bin lmc-adapter-argosbrain

# 1. Generate v0.2 scenarios
python3 tools/generate_v2_kubernetes_scenarios.py
cp /tmp/lmc2_gen/kubernetes-v2.json scenarios/kubernetes-v2.json

# 2. Pin oracles
python3 tools/oracle_pin_kubernetes.py
cp /tmp/lmc2_gen/kubernetes-v3.json scenarios/kubernetes-v3.json

# 3. Run any adapter against v3
./target/release/lmc-runner \
    --adapter ./target/release/lmc-adapter-argosbrain \
    --adapter-args "--corpus corpora/_work/kubernetes/kubernetes.argosbundle" \
    --scenarios scenarios/kubernetes-v3.json \
    --out results/your-adapter-kubernetes-v3-YYYY-MM-DD.json
```

## 6. What still doesn't measure

- **Multi-adapter pin convergence**: pinning against ArgosBrain is one
  data point. A more robust v0.4 would pin against a *consensus* of
  multiple structural adapters and treat any of those top-3 as a
  pass. We don't have enough non-ArgosBrain structural adapters yet.
- **Edit-cascade invalidation**: RereadCascade is still repeated reads
  without intermediate edits. Real edit cascades need runner support
  for an `op: "edit"` between queries.
- **Multimodal scenarios**: out of scope for v0.3, planned for v0.5+.
